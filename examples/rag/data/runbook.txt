RUNBOOK — Platform Operations
Last updated: 2024-03-01

---

INCIDENT: Database connection timeouts
Symptoms: API returning 503, logs show "connection timeout after 30s"
Steps:
  1. Check DB connection pool: SELECT count(*), state FROM pg_stat_activity GROUP BY state;
  2. If pool exhausted, identify long-running queries: SELECT pid, now() - query_start AS duration, query FROM pg_stat_activity WHERE state = 'active' ORDER BY duration DESC;
  3. Kill blocking queries if needed: SELECT pg_terminate_backend(pid) WHERE pid = <pid>;
  4. If DB unreachable, check replica lag and promote a replica if primary is down.
  5. Alert #oncall if not resolved within 10 minutes.

---

INCIDENT: Payment gateway timeouts (504 on /api/checkout)
Symptoms: Checkout returning 504, worker logs show "gateway timeout"
Steps:
  1. Check gateway status page.
  2. If gateway is up, check outbound network from API pods: kubectl exec -it <pod> -- curl -v https://gateway.example.com/health
  3. Manually retry failed orders from DLQ: kubectl exec -it worker-pod -- ./cli retry-dlq --job payment:charge
  4. Notify affected users if downtime exceeds 15 minutes.

---

INCIDENT: High memory usage (>80%)
Symptoms: WARN logs "High memory usage detected", potential OOMKills
Steps:
  1. Identify memory consumers: kubectl top pods -n api
  2. Check for memory leaks in recent deploys: compare current vs previous release with `helm diff`
  3. Restart affected pods if usage >90%: kubectl rollout restart deployment/api
  4. Scale up if issue persists: kubectl scale deployment/api --replicas=<n+1>

---

INCIDENT: Worker stuck / jobs not processing
Symptoms: Job queue growing, no worker activity in logs
Steps:
  1. Check worker pod status: kubectl get pods -n workers
  2. Look for crashloops: kubectl describe pod <pod>
  3. Check DLQ size: kubectl exec -it worker-pod -- ./cli dlq-stats
  4. Restart workers: kubectl rollout restart deployment/worker
  5. If queue is backed up >1000 jobs, scale workers temporarily.

---

DEPLOYMENT: Rolling back a bad release
Steps:
  1. Identify the last good revision: helm history <release>
  2. Roll back: helm rollback <release> <revision>
  3. Verify pods are healthy: kubectl rollout status deployment/<service>
  4. Check error rate in logs before closing the incident.

---

MAINTENANCE: Rotating secrets
Steps:
  1. Update secret in AWS Secrets Manager.
  2. Sync to Kubernetes: kubectl apply -f secrets/
  3. Restart affected deployments to pick up new values.
  4. Verify connections are healthy after restart.
